Part 1: State of the World 2026

1.1 软件开发的范式转移

关键数据: - 2025年，72%的开发者已在日常工作中使用AI辅助编码 (GitHub Survey) - 使用AI的开发者生产力提升 55-75% (McKinsey, 2025) - AI编写的代码占GitHub新增代码的 41% (GitHub Copilot Report)

核心洞察: > “You won’t be replaced by AI, but by an engineer using AI.” > — 软件工程的新现实

1.2 从”会写代码”到”会指挥AI”

传统开发者 AI时代开发者
手写每一行代码 设计Spec，让AI实现
记住API文档 描述需求，让AI查找
调试靠打断点 让AI解释+修复bug
独立完成任务 协调多个AI Agent

新能力要求: 1. Prompt Engineering → 如何精确描述需求 2. Context Engineering → 如何组织信息让AI理解 3. Agent Orchestration → 如何协调多个AI完成复杂任务 4. AI Judgment → 知道何时信任AI，何时人工干预

1.3 “Specs Are the New Source Code”

“在AI时代，详细的需求说明(Spec)比代码本身更重要。因为AI可以根据好的Spec生成代码，但无法从模糊的想法中猜出你要什么。” — Ravi Mehta, 前Facebook产品VP

实践建议: - 花更多时间写清楚你要什么 - 花更少时间纠结怎么实现 - 用结构化格式描述需求（Markdown, YAML） - 为项目维护一个 SPEC.md 文档

Part 2: 大模型的三阶段训练

2.1 为什么MBA需要了解这个？

因为训练方式决定了模型的能力边界和行为模式。理解训练过程，才能： - 知道模型擅长什么、不擅长什么 - 理解为什么有些prompt有效、有些无效 - 预测模型在不同场景下的表现

2.3 每个阶段的商业含义

阶段 成本 时间 谁在做 商业影响
Pretraining $10M-$1B 数月 少数大厂 决定基础能力上限
SFT $10K-$1M 数周 更多公司可做 决定任务适配性
RLHF $100K-$10M 数周 需要人类标注 决定用户体验

关键洞察: - 创业公司通常从Stage 2介入（基于开源基座模型微调） - 企业应用主要在Stage 2-3发力（私有数据微调+对齐） - Pretraining是大厂的游戏（资本密集型）

2.4 RLHF: 为什么ChatGPT”感觉”更好用？

RLHF (Reinforcement Learning from Human Feedback) 是让模型”更懂人”的关键：

用户问题 → 模型生成多个回答 → 人类标注员排序 → 训练奖励模型 → 强化学习优化

RLHF带来的改变: - ❌ 之前: 模型只追求”预测准确” - ✅ 之后: 模型追求”人类满意”

商业启示: - 用户反馈是最宝贵的数据资产 - “感觉好用”可能比”技术领先”更重要 - 收集偏好数据是建立护城河的方式之一

Part 3: 不是所有Token都平等

3.1 “中间迷失” (Lost in the Middle)

现象: 当上下文很长时，模型对开头和结尾的信息记得最清楚，中间的信息容易被忽略。

研究数据 (Liu et al., 2023): - 信息在开头: 准确率 75% - 信息在中间: 准确率 45% ⚠️ - 信息在结尾: 准确率 70%

3.2 实践建议

把重要信息放在对的位置:

信息类型 建议位置 原因
系统指令 最开头 设定基调
关键约束 开头 + 结尾重复 强化记忆
背景资料 中间 可被部分遗忘
具体问题 最结尾 紧邻输出

Prompt结构模板:

# 系统指令 (开头 - 最重要)
你是一个专业的财务分析师...

# 背景资料 (中间 - 可以长)
以下是公司的财务报表...
[大量数据]

# 关键约束提醒 (结尾前 - 重复强调)
请记住：
- 所有数字必须有来源
- 不确定时说"需要更多信息"

# 具体问题 (最后 - 紧邻输出)
请分析该公司的现金流状况。

3.3 Context Rot: 长对话的衰减

现象: 在长对话中，早期的信息会逐渐”腐烂”——模型越来越难以准确回忆和使用它们。

Chroma Research发现: - 10轮对话后，早期信息召回准确率下降 40% - 某些细节可能被”覆盖”或”扭曲”

应对策略: 1. 定期总结: 每隔几轮让AI总结关键信息 2. 外部记忆: 重要信息存储在外部系统（RAG） 3. 主动提醒: 在关键时刻重新注入重要上下文

Part 4: 上下文工程 (Context Engineering)

4.1 定义

上下文工程 = 设计和管理输入给AI的信息，使其能够最有效地完成任务的艺术与科学。

这不仅仅是”写好prompt”，而是一整套系统性的方法论：

┌─────────────────────────────────────────────────┐
│              上下文工程的四个维度                 │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. 信息选择 (What)                              │
│     哪些信息应该放入上下文？                      │
│                                                 │
│  2. 信息组织 (How)                               │
│     信息以什么格式、顺序呈现？                    │
│                                                 │
│  3. 信息压缩 (Compression)                       │
│     如何在有限窗口内放入最多有用信息？            │
│                                                 │
│  4. 信息更新 (Update)                            │
│     随着对话进行，如何动态调整上下文？            │
│                                                 │
└─────────────────────────────────────────────────┘

4.2 Manus团队的教训

Manus是2025年爆火的AI Agent产品，他们的上下文工程经验值得学习：

四次架构重构: 1. V1: 简单拼接所有信息 → 经常超出窗口限制 2. V2: 只保留最近N轮 → 丢失重要早期信息 3. V3: LLM自动摘要 → 摘要丢失关键细节 4. V4: 分层记忆 + 按需检索 → 效果最好

核心教训: > “上下文工程是一门实验科学，没有银弹。你必须根据具体任务不断迭代。” > — Manus团队博客

4.3 实用技巧清单

技巧 说明 适用场景

Part 5: AI开发工具生态

5.1 AI IDE: Cursor & Windsurf

Cursor (cursor.com): - 基于VSCode的AI-native编辑器 - 核心功能: Cmd+K 自然语言编辑, Cmd+L AI对话 - 杀手特性: .cursorrules 项目级上下文配置

Windsurf (codeium.com/windsurf): - Codeium出品的AI IDE - 特点: 更激进的AI集成，“Flow”模式 - 适合: 喜欢AI主导的开发风格

配置示例 (.cursorrules):

# 项目上下文配置
project_type: "Python FastAPI backend"
coding_style: "Google Python Style Guide"
always_include:
  - "README.md"
  - "docs/API.md"
preferences:
  - "使用type hints"
  - "函数必须有docstring"
  - "错误处理用自定义异常类"

5.2 Modern Terminal: Warp

Warp (warp.dev): - AI增强的现代终端 - 核心功能: 自然语言生成命令、智能补全 - 适合: 不熟悉命令行但需要使用的人

使用示例:

# 用自然语言描述
> "找出当前目录下最大的10个文件"

# Warp自动生成
$ find . -type f -exec du -h {} + | sort -rh | head -10

5.3 Claude Code 深度解析

Claude Code 是Anthropic官方的AI编程Agent，官方用法值得深入研究。

核心概念: 1. System Prompts: 定义Agent的角色和能力边界 2. System Reminders: 在对话中动态注入的提醒 (<system-reminder>标签) 3. Sub-agent Spawning: 复杂任务分解给子Agent

最佳实践 (来自Anthropic内部): - 维护 CLAUDE.md 项目文档 - 使用结构化的任务描述 - 设置明确的成功标准 - 让Claude Code自己决定实现细节
