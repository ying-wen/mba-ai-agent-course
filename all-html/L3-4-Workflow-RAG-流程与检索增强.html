Part 1：从对话到工作流

单轮对话的局限

用户: "帮我分析一下竞争对手"
AI:   [给出泛泛而谈的分析]

问题:
- 缺乏具体数据支撑
- 无法获取实时信息
- 分析深度有限

工作流的价值

用户: "帮我分析一下竞争对手"

工作流自动执行:
1. 搜索引擎获取最新新闻
2. 数据库查询历史财务数据
3. 调用分析模板结构化输出
4. 交叉验证关键信息
5. 生成最终报告

输出: 有数据支撑、结构清晰、可追溯的分析报告

MBA视角：工作流把AI从”聊天工具”变成”生产力系统”。单轮对话是问答，工作流是交付。

Part 2：LangChain简介

什么是LangChain？

LangChain是构建LLM应用的开发框架，类似于Web开发中的Django/Rails。

核心价值： - 统一不同LLM的调用方式 - 预置记忆、检索、工具模块 - 将多个组件串联成工作流

核心组件

组件 功能 类比
Models 调用各种LLM 数据库驱动
Chains 串联多个步骤 函数管道
Memory 保存对话历史 Session
Retrievers 检索相关文档 搜索引擎
Tools 调用外部API 第三方服务

Part 3：RAG核心概念

为什么需要RAG？

LLM有两个根本局限：

  1. 知识截止：训练数据有时间截止点，不知道最新信息
  2. 幻觉问题：可能生成看似合理但错误的内容

RAG的解决思路：先检索，再生成

RAG工作流程

┌─────────────────────────────────────────────────────┐
│                     RAG 流程                         │
├─────────────────────────────────────────────────────┤
│                                                     │
│  离线阶段 (索引构建)                                │
│  ─────────────────────                              │
│  文档 → 分块 → 向量化 → 存入向量数据库              │
│                                                     │
│  在线阶段 (查询响应)                                │
│  ─────────────────────                              │
│  用户问题 → 向量化 → 相似度搜索 → 获取相关片段      │
│      ↓                                              │
│  [问题 + 相关片段] → LLM → 答案                    │
│                                                     │
└─────────────────────────────────────────────────────┘

用一个例子说明

场景：企业内部知识库问答

文档：公司员工手册.pdf (100页)

离线处理:
1. 把手册分成200个小块（每块约500字）
2. 每块转成向量（一串数字）
3. 存入向量数据库

用户提问: "公司的年假政策是什么？"

在线处理:
1. 问题转成向量
2. 在数据库中找最相似的5个块
3. 把问题+这5个块一起给LLM
4. LLM基于这些内容回答

输出: "根据员工手册第3章，正式员工每年享有15天带薪年假..."

Part 4：分块策略

为什么分块重要？

分块（Chunking）是RAG效果的关键。块太大，检索不精准；块太小，丢失上下文。

常见分块策略

策略 说明 适用场景
固定大小 每500字切分 通用场景
句子边界 按句号/段落切分 文章类
语义分块 按语义相似度切分 高质量要求
递归分块 多级粒度 长文档
父子分块 小块检索+大块上下文 平衡精度和上下文

分块最佳实践

  1. 重叠窗口：相邻块之间保留20%重叠，避免信息断裂
  2. 保留元数据：记录每块来源（文档名、页码）
  3. 测试迭代：没有万能参数，需要针对具体数据调优

Part 5：向量数据库

什么是向量数据库？

传统数据库用精确匹配（WHERE name = ‘John’），向量数据库用相似度匹配。

传统数据库: "年假" → 只能找到包含"年假"的文档
向量数据库: "年假" → 能找到"休假"、"假期"、"带薪休息"等语义相似的内容

主流选型

数据库 类型 特点 适用场景
Pinecone 云服务 托管、易用 快速上手
Milvus 开源 高性能、可扩展 大规模生产
Chroma 开源 轻量、嵌入式 原型开发
Weaviate 开源 多模态支持 图文混合
pgvector 扩展 基于PostgreSQL 已有PG环境

Embedding模型推荐

模型 厂商 特点 维度
text-embedding-3-large OpenAI 综合最强 3072
BGE-M3 BAAI 多语言、开源 1024
Jina-embeddings-v3 Jina AI 长文本支持 1024
GTE-large 阿里 中文优化 1024

Part 6：RAG优化策略

常见问题与解决方案

问题 原因 解决方案
检索不准 分块粒度不对 调整chunk size
召回不全 只用向量检索 混合检索（关键词+向量）
答案错误 LLM没用检索结果 优化Prompt强调”基于以下内容”
速度太慢 检索量太大 预过滤、分层检索

进阶技术

技术 说明 效果
重排序(Rerank) 用更精细的模型对检索结果重排 精度+20%
Self-RAG LLM自己判断是否需要检索 减少不必要检索
RAPTOR 递归摘要构建层次索引 长文档效果提升

Part 7：Coze平台实战

什么是Coze？

Coze（扣子）是字节跳动的AI应用开发平台，特点是低代码、可视化，适合非技术人员快速搭建AI应用。

核心概念

概念 说明
Bot 一个完整的AI应用
Workflow 多步骤执行流程
Plugin 可调用的外部能力
Knowledge 知识库（RAG）
Memory 对话记忆

Workflow节点类型

节点 功能 示例
LLM 调用大模型 生成、分析、总结
Code 执行代码 数据处理、计算
Knowledge 检索知识库 RAG
Plugin 调用外部API 搜索、天气
Condition 条件判断 分支逻辑
Loop 循环执行 批量处理

实战案例

案例1：智能客服Bot

用户问题 → 知识库检索 → LLM生成回复 → 满意度判断
                                           ↓
                                    不满意 → 转人工

案例2：日报生成助手

定时触发 → 搜索行业新闻 → LLM筛选重要内容 → 格式化输出 → 发送邮件

案例3：面试助手

上传简历 → 解析关键信息 → 生成面试问题 → 评估回答 → 输出评分

常见陷阱

  1. 数据质量被低估：垃圾进垃圾出
  2. 分块策略一刀切：不同类型文档需要不同策略
  3. 忽视更新机制：知识库会过时
  4. 没有评估体系：不知道效果好不好
