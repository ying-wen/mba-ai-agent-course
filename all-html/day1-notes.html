1.1 什么是大语言模型？

通俗理解

想象一个读过人类所有书籍、文章、代码的”超级读者”。它不是真正理解内容，而是学会了语言的统计规律——给定前文，预测下一个最可能出现的词。

输入: "今天天气真"
模型预测: "好" (概率85%) / "糟" (概率10%) / "热" (概率5%)

这个简单的”预测下一个词”任务，在足够大的数据和模型规模下，涌现出了惊人的能力：推理、创作、编程、翻译……

Transformer架构（非技术人员版）

不需要理解数学细节，只需知道Transformer的三个关键创新：

  1. 注意力机制 (Attention): 模型可以”关注”输入中任意位置的信息，不受距离限制
  2. 并行计算: 可以同时处理整个序列，训练速度大幅提升
  3. 可扩展性: 架构简洁，容易通过增加参数获得更强能力

MBA视角: Transformer的发明者大多已离开Google创业。这项技术创造了万亿美元的新市场，但发明者的直接回报有限——这是一个关于创新价值分配的经典案例。

1.2 Token经济学

什么是Token？

Token是LLM处理文本的基本单位，不是字符，也不是单词，而是介于两者之间的”子词”。

"Hello, world!" → ["Hello", ",", " world", "!"]  (4 tokens)
"人工智能" → ["人工", "智能"]  (2 tokens, 中文通常1-2字=1token)

为什么Token重要？

  1. 定价依据: API按token计费
  2. 上下文限制: 每个模型有token上限
  3. 成本估算: 1000 tokens ≈ 750英文单词 ≈ 500中文字

1.3 Prompt工程

为什么Prompt重要？

同一个模型，不同的提示词可能导致完全不同的输出质量。Prompt工程是零成本提升AI效果的最有效方法。

核心技巧

1. 角色设定 (Role Prompting)

❌ 普通: "分析一下特斯拉的财报"

✅ 优化: "你是一位有15年经验的华尔街分析师，专注于新能源汽车行业。
         请从以下角度分析特斯拉2025Q4财报：
         1. 营收增长驱动因素
         2. 毛利率变化原因
         3. 与市场预期的差异
         4. 投资建议"

2. 结构化输出

请用以下JSON格式输出分析结果：
{
  "summary": "一句话总结",
  "key_metrics": [
    {"name": "指标名", "value": "数值", "yoy_change": "同比变化"}
  ],
  "risks": ["风险1", "风险2"],
  "recommendation": "买入/持有/卖出"
}

3. 思维链 (Chain of Thought)

❌ 直接问: "这个创业项目值得投资吗？"

✅ 引导推理: "请一步步分析这个创业项目：
              第一步：市场规模和增长潜力
              第二步：竞争格局和壁垒
              第三步：团队背景和执行力
              第四步：商业模式和盈利路径
              第五步：综合以上分析，给出投资建议"

2.1 从对话到工作流

单轮对话的局限

用户: "帮我分析一下竞争对手"
AI:   [给出泛泛而谈的分析]

问题:
- 缺乏具体数据支撑
- 无法获取实时信息
- 分析深度有限

工作流的价值

用户: "帮我分析一下竞争对手"

工作流自动执行:
1. 搜索引擎获取最新新闻
2. 数据库查询历史财务数据
3. 调用分析模板结构化输出
4. 交叉验证关键信息
5. 生成最终报告

输出: 有数据支撑、结构清晰、可追溯的分析报告

2.2 LangChain核心概念

什么是LangChain？

LangChain是构建LLM应用的开发框架，类似于Web开发中的Django/Rails。它提供：

  • 标准化抽象: 统一不同LLM的调用方式
  • 组件库: 预置的记忆、检索、工具模块
  • 编排能力: 将多个组件串联成工作流

2.3 RAG: 检索增强生成

为什么需要RAG？

LLM的两个根本局限：

  1. 知识截止: 训练数据有时间截止点
  2. 幻觉问题: 可能生成看似合理但错误的内容

RAG的解决思路：先检索，再生成

定义

3.2 ReAct框架

论文背景

ReAct (Reasoning + Acting) 是2022年Google Research和Princeton提出的Agent框架，将推理和行动交织进行。

核心思想

传统方法: 
  思考 → 思考 → 思考 → 行动

ReAct:
  思考 → 行动 → 观察 → 思考 → 行动 → 观察 → ...

实际示例

用户任务: "帮我查一下苹果公司最新的市值，并与微软比较"

Agent执行过程:

Thought 1: 我需要查询苹果公司的当前市值
Action 1: search("Apple Inc market cap 2026")
Observation 1: Apple Inc. (AAPL) has a market cap of $3.2 trillion...

Thought 2: 现在我需要查询微软的市值
Action 2: search("Microsoft market cap 2026")
Observation 2: Microsoft Corporation (MSFT) market cap is $3.4 trillion...

Thought 3: 我已经有了两家公司的数据，可以进行比较
Action 3: 生成比较分析

Final Answer:
截至2026年2月，苹果市值约3.2万亿美元，微软市值约3.4万亿美元。
微软市值超过苹果约2000亿美元，差距约6%...

为什么ReAct有效？

  1. 透明性: 每一步推理可追溯
  2. 可控性: 可以中断/调整执行
  3. 准确性: 基于实际观察而非臆测
  4. 可调试: 出错时容易定位问题

3.3 反思机制 (Reflection)

为什么需要反思？

Agent执行过程中可能出错： - 工具调用失败 - 获取的信息不准确 - 推理逻辑有漏洞

反思机制让Agent能够自我评估并修正。

3.4 规划算法

为什么需要规划？

复杂任务需要分解成子任务，按顺序或并行执行。好的规划能够： - 提高执行效率 - 减少无效尝试 - 处理任务依赖

常见规划方法

Tree of Thoughts (ToT)

                    问题
                      │
           ┌─────────┼─────────┐
           │         │         │
         思路A     思路B     思路C
           │         │         │
        ┌──┴──┐   ┌──┴──┐   ┌──┴──┐
        │     │   │     │   │     │
       A1    A2  B1    B2  C1    C2
        ↓         ↓    
      评估      评估       (剪枝)
        ↓         ↓
     选择最优路径
  • 思路: 探索多个可能的推理路径
  • 优点: 找到更优解
  • 缺点: 计算成本高

Graph of Thoughts (GoT)

  • 允许思维节点之间形成图结构
  • 支持合并、分裂思路
  • 更灵活但更复杂

实际选择建议

任务复杂度 推荐方法 说明
简单 直接执行 ReAct足够
中等 ToT 关键决策点展开
复杂 分层规划 先粗后细
超复杂 人机协作 Agent提建议，人做决策

4.2 工具使用 (Tool Use)

Function Calling

Function Calling是让LLM调用外部函数/API的标准机制：

┌─────────────────────────────────────────────────────┐
│              Function Calling 流程                   │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 定义函数                                        │
│  {                                                  │
│    "name": "get_weather",                          │
│    "description": "获取城市天气",                  │
│    "parameters": {                                  │
│      "type": "object",                             │
│      "properties": {                               │
│        "city": {"type": "string"}                  │
│      }                                              │
│    }                                                │
│  }                                                  │
│                                                     │
│  2. 用户请求                                        │
│  "上海今天天气怎么样？"                            │
│                                                     │
│  3. LLM生成函数调用                                │
│  {"name": "get_weather", "arguments": {"city": "上海"}}
│                                                     │
│  4. 应用执行函数，返回结果                         │
│  {"temperature": 22, "condition": "晴"}            │
│                                                     │
│  5. LLM根据结果生成回复                            │
│  "上海今天天气晴朗，气温22度，适合外出。"          │
│                                                     │
└─────────────────────────────────────────────────────┘

常见工具类型

类型 示例 用途

代码 Python解释器 复杂计算、数据处理

文件 读写本地文件 文档处理

工具安全

风险 说明 缓解措施
注入攻击 恶意输入导致非预期调用 输入验证、沙盒执行
权限滥用 Agent获取过多权限 最小权限原则
数据泄露 敏感信息被发送到外部 数据脱敏、访问控制
成本失控 无限循环调用 调用次数限制、超时

企业工具集成最佳实践

  1. 渐进式授权: 从只读开始，逐步开放写入
  2. 审计日志: 记录所有工具调用
  3. 人在回路: 高风险操作需人工确认
  4. 降级策略: 工具不可用时的备选方案
