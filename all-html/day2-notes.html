5.1 为什么需要多Agent？

单Agent的局限

单Agent处理复杂任务:

用户: "帮我做一个完整的市场调研报告"

单Agent尝试:
├── 搜索市场数据 ✓
├── 分析竞争格局 ✓
├── 预测市场趋势 △ (开始疲劳)
├── 撰写执行摘要 △ (质量下降)
├── 设计数据图表 ✗ (上下文溢出)
└── 最终审核校对 ✗ (已经混乱)

问题:
• 上下文窗口不足
• 单一视角，缺乏交叉验证
• 角色切换导致质量不稳定
• 无法并行处理

多Agent的优势

多Agent协作:

用户: "帮我做一个完整的市场调研报告"

Coordinator (协调者)
    │
    ├── Researcher (研究员)
    │   └── 负责数据收集和分析
    │
    ├── Analyst (分析师)
    │   └── 负责竞争格局和趋势
    │
    ├── Writer (撰稿人)
    │   └── 负责报告撰写
    │
    └── Reviewer (审核员)
        └── 负责质量把控

优势:
• 专业分工，各司其职
• 独立上下文，不会溢出
• 可并行执行
• 交叉验证，质量更高

何时使用多Agent？

场景 单Agent 多Agent
简单问答 ✅ 过度设计
单一技能任务 ✅ 不必要
需要多视角 ❌ ✅
复杂工作流 勉强 ✅
需要验证 ❌ ✅
长时间任务 不稳定 ✅

5.3 多Agent协作模式

模式1: 顺序执行 (Pipeline)

Agent A ──► Agent B ──► Agent C ──► 输出

示例: 研究 → 分析 → 撰写
优点: 简单清晰
缺点: 无法并行

模式2: 并行执行

        ┌── Agent A ──┐
输入 ───┼── Agent B ──┼──► 聚合 ──► 输出
        └── Agent C ──┘

示例: 多角度同时分析
优点: 效率高
缺点: 需要聚合策略

模式3: 层级结构

           Manager
              │
    ┌─────────┼─────────┐
    │         │         │
 Worker A  Worker B  Worker C

示例: CEO → 部门经理 → 执行者
优点: 可扩展、职责清晰
缺点: 通信开销

模式4: 辩论式

Agent A ◄───辩论───► Agent B
    │                   │
    └───────┬───────────┘
            ▼
         裁判Agent
            │
            ▼
          结论

示例: 正反方辩论，得出更全面结论
优点: 避免单一视角偏见
缺点: 成本翻倍

第11-12课时: MCP与Skill生态

6.1 MCP协议核心

设计背景

2024年11月，Anthropic发布MCP (Model Context Protocol)，旨在解决：

问题: 每个AI应用都要单独集成外部工具，造成大量重复工作

解决方案: 标准化协议，一次集成，到处使用

类比理解

MCP 类似于 USB 协议

没有USB之前:
• 每种设备有专属接口
• 鼠标、键盘、打印机各不相同
• 兼容性差，成本高

有了USB之后:
• 统一接口标准
• 任何设备即插即用
• 生态繁荣

MCP对AI工具集成的作用 = USB对硬件的作用

架构三层模型

层级 组件 职责
Host Claude Desktop, IDE 管理生命周期、安全策略
Client MCP Client 与Server通信
Server GitHub MCP, Slack MCP 暴露工具能力

四大能力

  1. Resources: 暴露数据供LLM读取
  2. Tools: 可执行的函数/API
  3. Prompts: 预定义的提示词模板
  4. Sampling: Server请求LLM生成 (高级)

6.2 Skill系统

什么是Skill？

OpenClaw的能力扩展模块，核心是SKILL.md——一个让Agent学习如何使用工具的说明文档。

与传统Plugin的区别

传统Plugin:
代码 ──► 代码接口 ──► 执行

Skill:
SKILL.md ──► LLM阅读理解 ──► LLM决定如何调用 ──► 执行

关键区别: LLM自己读文档学习使用方法！

为什么这样设计更好？

  1. 灵活性: 不需要预定义所有调用方式
  2. 容错性: LLM可以处理模糊指令
  3. 可扩展: 增加能力只需写文档
  4. 可维护: 非程序员也能创建Skill

6.3 MCP vs Skill

维度 MCP Server Skill
标准化 JSON-RPC协议 无标准
跨平台 任何支持MCP的Host 仅OpenClaw
实现复杂度 中等 低
适用场景 通用服务集成 特定场景快速实现

选择建议

  • 需要跨平台 → MCP
  • 仅OpenClaw使用 → Skill更简单
  • 复杂企业集成 → MCP (标准化、可维护)
  • 快速原型 → Skill (1小时上线)

8.2 行业应用深度案例

金融: AI投研

痛点: - 分析师信息过载 - 研报同质化严重 - 时效性要求高

Agent方案:

数据Agent → 分析Agent → 撰写Agent → 合规Agent
    ↓            ↓            ↓            ↓
 实时数据     深度分析     结构化报告   风险审核

效果: - 研报初稿: 4小时→30分钟 - 覆盖标的: +300% - 合规问题: -80%

法律: 合同审查

痛点: - 合同条款繁多 - 人工审查耗时 - 风险条款易遗漏

Agent方案:

┌─────────┐    ┌─────────┐    ┌─────────┐
│合同解析 │───►│条款比对 │───►│风险标注 │
│ Agent   │    │ Agent   │    │ Agent   │
└─────────┘    └─────────┘    └─────────┘
     │              │              │
     ▼              ▼              ▼
  结构化提取    vs标准模板     风险等级+建议

代表公司: Harvey AI (估值$1.5B)

效果: - 审查时间: 2小时→15分钟 - 风险识别率: +40% - 律师效率: 3x

医疗: 诊断辅助

应用场景: - 影像诊断辅助 - 病历摘要生成 - 用药建议审核

合规要点: - 必须是”辅助”而非”替代” - 需要医生确认 - 患者知情同意 - 数据隐私保护

代表产品: Google Med-PaLM, 百度灵医

8.3 Agent安全与治理

风险矩阵

风险类型 说明 缓解措施
幻觉 生成错误信息 RAG验证、人工审核
越权 执行未授权操作 权限控制、审批流程
数据泄露 敏感信息外传 数据脱敏、访问控制
偏见 输出存在歧视 测试审计、多样性训练
成本失控 API费用超支 预算限制、监控告警
依赖风险 供应商中断 多供应商、降级方案
