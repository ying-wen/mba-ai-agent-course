## 为什么这节课是核心？

如果说前面几节课讲的是"如何让AI回答问题"，这节课讲的是"如何让AI完成任务"。

这是一个根本性的跨越：

| 维度 | 聊天机器人 | AI Agent |
|------|------------|----------|
| 交互模式 | 问答式 | 任务式 |
| 决策能力 | 被动响应 | 主动规划 |
| 工具使用 | 无/有限 | 丰富且动态 |
| 任务范围 | 单轮完成 | 多步骤复杂任务 |
| 错误处理 | 简单重试 | 反思改进 |

> **MBA视角**：Agent是AI从"工具"变成"员工"的关键技术。理解Agent，就理解了AI如何在组织中承担实际工作。

## Part 1：什么是AI Agent？

### 定义

AI Agent是一个**能自主决策、使用工具、完成复杂任务**的AI系统。

用一个类比来理解：

**普通LLM = 百科全书**
- 你问什么，它查什么
- 不会主动做事

**AI Agent = 远程员工**
- 你给任务，它自己规划怎么做
- 需要时自己查资料、用工具
- 遇到问题自己想办法

### Agent架构总览

![Agent架构总览](../slides-marp/assets/images/lilian-weng-agent-overview.png)
*来源: Lilian Weng, "LLM Powered Autonomous Agents" (2023)*

上图展示了Agent的核心架构：
- **LLM**作为"大脑"进行规划和推理
- **Memory**提供短期和长期记忆能力
- **Tools**让Agent能与外部世界交互
- **Planning**让Agent能分解复杂任务

### Agent的四要素

就像一个员工需要具备的能力：

| 要素 | 能力 | 商业类比 |
|------|------|----------|
| **感知 (Perceive)** | 理解用户意图、获取环境信息 | 理解老板需求 |
| **思考 (Think)** | 分析问题、制定方案 | 制定工作计划 |
| **行动 (Act)** | 调用工具、执行操作 | 执行具体任务 |
| **反思 (Reflect)** | 评估结果、改进策略 | 复盘总结经验 |

### Agent的核心循环

Agent不是一次性输出，而是**循环执行**直到任务完成：

```
感知 → 思考 → 行动 → 观察结果 → 反思 → 继续思考 → ...
```

这个循环会一直进行，直到：
- 任务完成
- 达到最大步数
- 用户中断

## Part 2：ReAct框架

### 论文背景

ReAct (Reasoning + Acting) 是2022年Google Research和Princeton提出的Agent框架，将**推理**和**行动**交织进行。

### ReAct执行示例

![ReAct框架示例](../slides-marp/assets/images/lilian-weng-react.png)
*来源: Lilian Weng, "LLM Powered Autonomous Agents" (2023)*

上图展示了ReAct的实际执行过程：
- **Thought**：Agent的推理过程
- **Action**：Agent调用的工具
- **Observation**：工具返回的结果

### 核心思想

传统方法把思考和行动分开：
```
思考 → 思考 → 思考 → 行动
```

ReAct把它们交织在一起：
```
思考 → 行动 → 观察 → 思考 → 行动 → 观察 → ...
```

### 为什么这样更好？

想象你在做市场调研：

**传统方法**（先想完再做）：
```
1. 我觉得市场规模大概是100亿
2. 我觉得主要竞争对手是A、B、C
3. 我觉得增长率大概是15%
4. 开始写报告...

问题：全凭猜测，可能全错
```

**ReAct方法**（边做边想）：
```
1. 我需要查市场规模 → 搜索"中国咖啡市场规模2026"
2. 观察：搜索结果显示市场规模是850亿
3. 我需要查竞争格局 → 搜索"中国咖啡连锁品牌排名"
4. 观察：瑞幸第一、星巴克第二...
5. 基于真实数据写报告

优势：每一步都基于真实信息
```

### 实际执行示例

```
用户任务: "帮我查一下苹果公司最新的市值，并与微软比较"

Agent执行过程:

Thought 1: 我需要查询苹果公司的当前市值
Action 1: search("Apple Inc market cap 2026")
Observation 1: Apple Inc. (AAPL) has a market cap of $3.2 trillion...

Thought 2: 现在我需要查询微软的市值
Action 2: search("Microsoft market cap 2026")
Observation 2: Microsoft Corporation (MSFT) market cap is $3.4 trillion...

Thought 3: 我已经有了两家公司的数据，可以进行比较了
Action 3: 生成比较分析

Final Answer:
截至2026年2月，苹果市值约3.2万亿美元，微软市值约3.4万亿美元。
微软市值超过苹果约2000亿美元，差距约6%...
```

### ReAct的优势

| 优势 | 说明 |
|------|------|
| **透明性** | 每一步推理可追溯，出问题知道在哪 |
| **可控性** | 可以中途干预、调整方向 |
| **准确性** | 基于实际观察而非臆测 |
| **可调试** | 出错时容易定位问题 |

## Part 3：反思机制

### 为什么需要反思？

Agent执行过程中可能出错：
- 工具调用失败
- 获取的信息不准确
- 推理逻辑有漏洞
- 结果不符合预期

没有反思的Agent就像一个不会总结经验的员工——同样的错误会反复犯。

### Reflexion机制

![Reflexion反思机制](../slides-marp/assets/images/lilian-weng-reflexion.png)
*来源: Lilian Weng, "LLM Powered Autonomous Agents" (2023)*

上图展示了Reflexion的工作原理：
- Agent执行任务后评估结果
- 如果失败，生成反思摘要
- 反思存入记忆，指导下次执行

### 三种反思策略

#### 1. Self-Refine（自我改进）

```
生成初稿 → 自我评估 → 改进 → 再评估 → ...

示例：
初稿："市场规模100亿"
评估："没有数据来源，可信度低"
改进："根据艾瑞咨询报告，2025年市场规模为95亿元[1]"
```

**优点**：简单、成本低
**缺点**：可能陷入局部最优（自己看不到自己的问题）

#### 2. Reflexion（反思记忆）

```
失败经验 → 提取教训 → 存入记忆 → 下次避免

示例：
失败："搜索'苹果市值'返回了水果苹果的结果"
教训："搜索公司信息时要加'公司'或股票代码"
记忆：存储这条教训
下次：搜索"苹果公司AAPL市值"
```

**优点**：持续学习、避免重复错误
**缺点**：需要维护记忆系统

#### 3. Critic Agent（批评者Agent）

```
执行Agent完成任务 → Critic Agent评估 → 反馈改进建议

类似于：
- 员工写报告 → 主管审核 → 修改意见 → 员工修改
```

**优点**：视角独立、更客观
**缺点**：成本翻倍、需要协调两个Agent

### 如何选择反思策略？

| 场景 | 推荐策略 | 原因 |
|------|----------|------|
| 简单任务 | Self-Refine | 成本低，够用 |
| 需要积累经验 | Reflexion | 持续改进 |
| 高风险决策 | Critic Agent | 需要双重检验 |
| 成本敏感 | Self-Refine | 最便宜 |

## Part 4：规划算法

### 为什么需要规划？

复杂任务需要分解成子任务，好的规划能够：
- 提高执行效率
- 减少无效尝试
- 处理任务依赖

### 核心范式：Plan-and-Execute

**核心思想**："三思而后行"

与ReAct的"边做边想"不同，Plan-and-Execute要求Agent在执行前先生成完整的计划。

**工作流程**：
1. **Plan**：理解任务，分解为子任务清单
2. **Solve**：逐个执行子任务，直到全部完成

**适用场景**：
- 任务步骤清晰（如"写一份包含摘要、正文、结论的报告"）
- 步骤之间依赖性强（必须先查A，再查B，才能对比A和B）
- 需要避免中途跑偏

**Prompt示例**：
```
请解决这个问题：{问题}

首先，制定一个详细的步骤计划。
然后，按照计划一步步执行。
每完成一步，打印当前进度。
```

### Tree of Thoughts (ToT)

ToT的核心思想：**不要只走一条路，同时探索多条路**

```
                    问题
                      │
           ┌─────────┼─────────┐
           │         │         │
         思路A     思路B     思路C
           │         │         │
        ┌──┴──┐   ┌──┴──┐   (剪枝)
        │     │   │     │   
       A1    A2  B1    B2  
        ↓         ↓    
      评估      评估  
        ↓         
     选择最优
```

**示例**：解决一个商业问题

```
问题：如何提升用户留存？

思路A：优化产品体验
├── A1：简化注册流程
└── A2：增加新手引导

思路B：增强用户粘性
├── B1：增加社交功能
└── B2：设计积分体系

思路C：提高服务质量
└── （评估后认为不是核心问题，剪枝）

评估每个方案的可行性和ROI，选择最优
```

### 规划方法选择建议

| 任务复杂度 | 推荐方法 | 说明 |
|------------|----------|------|
| 简单 | 直接执行 | ReAct足够 |
| 中等 | ToT | 关键决策点展开多路径 |
| 复杂 | 分层规划 | 先粗后细 |
| 超复杂 | 人机协作 | Agent提建议，人做决策 |

## Part 5：Agent设计模式

### 模式1：单一Agent

```
用户 → Agent → 结果

适用：简单任务、快速响应
```

### 模式2：工具增强Agent

```
用户 → Agent ←→ 工具库 → 结果

工具库包括：
- 搜索引擎
- 计算器
- 数据库
- API调用
- 代码执行
```

### 模式3：多Agent协作

```
用户 → Coordinator Agent
              │
    ┌─────────┼─────────┐
    ↓         ↓         ↓
 Agent A   Agent B   Agent C
    │         │         │
    └─────────┴─────────┘
              ↓
           结果汇总

适用：复杂任务需要多种专业能力
```

### 设计模式决策树

```
任务复杂度如何？
    │
    ├── 简单（单步可完成）
    │   └── 单一Agent，无需工具
    │
    ├── 中等（需要外部信息/计算）
    │   └── 工具增强Agent
    │
    └── 复杂（需要多种专业能力）
        │
        └── 是否可以并行？
            │
            ├── 是 → 多Agent并行
            └── 否 → 多Agent串行
```

## Part 7：企业应用案例

### 案例1：AI投研助手

**背景**：某券商需要处理海量研报、新闻、财务数据

**Agent架构**：
```
PM Agent (项目经理)
    │
    ├── Data Agent (数据收集)
    │   └── 搜索、爬取、数据库查询
    │
    ├── Analyst Agent (分析研究)
    │   └── 财务分析、竞争格局
    │
    ├── Writer Agent (报告撰写)
    │   └── 结构化输出
    │
    └── Review Agent (合规审核)
        └── 风险检查、合规审核
```

**效果**：
- 研报初稿时间：4小时 → 30分钟
- 数据覆盖：+200%
- 合规问题：-80%

### 案例2：Klarna客服AI

**背景**：瑞典金融科技公司，每月数百万客服请求

**方案**：基于OpenAI的客服Agent，处理退款、查询、投诉等场景

**效果**（2024年数据）：
- 处理量：相当于700名全职客服
- 解决时间：11分钟 → 2分钟
- 客户满意度：持平（关键！）
- 年节省成本：$4000万

**争议与反思**：
- 引发裁员担忧
- "效率提升"vs"就业影响"的平衡
- 适合重复性高、标准化强的场景

## Part 8：Agent开发实践

### 30/60/90天路线图

**第一阶段（0-30天）：验证可行性**

- [ ] 选择1个具体业务场景
- [ ] 用现成工具（如Coze）搭建原型
- [ ] 收集100个真实用户反馈
- [ ] 计算粗略ROI

**第二阶段（30-60天）：小规模试点**

- [ ] 优化Prompt和工作流
- [ ] 接入必要的内部系统
- [ ] 建立评估指标体系
- [ ] 培训试点用户

**第三阶段（60-90天）：规模化落地**

- [ ] 完善异常处理
- [ ] 建立监控告警
- [ ] 制定SOP和培训材料
- [ ] 扩展到更多用户/场景

### 常见失败原因

| 失败模式 | 根因 | 解决方案 |
|----------|------|----------|
| Agent不知道该做什么 | 任务描述模糊 | 明确成功标准 |
| Agent做了不该做的事 | 权限控制不足 | 最小权限原则 |
| Agent陷入死循环 | 没有终止条件 | 设置最大步数 |
| Agent输出不稳定 | 缺少评估机制 | 建立质量门控 |
| 成本失控 | 没有预算限制 | 设置token上限 |

## 本讲总结

### 核心概念

- **Agent = 能自主决策、使用工具、完成复杂任务的AI系统**
- **ReAct = 推理和行动交织进行**
- **反思 = Agent自我评估和改进的能力**
- **规划 = 将复杂任务分解为可执行步骤**

### Agent vs 传统AI的本质区别

| 传统AI | Agent |
|--------|-------|
| 回答问题 | 完成任务 |
| 一次性输出 | 循环执行 |
| 被动响应 | 主动规划 |
| 固定能力 | 工具扩展 |

### MBA关键洞察

1. **Agent是AI从"工具"变成"员工"的技术**
2. **设计Agent = 设计一个虚拟岗位**
3. **Agent需要监督，就像新员工需要管理**
4. **从简单场景开始，逐步扩展**
